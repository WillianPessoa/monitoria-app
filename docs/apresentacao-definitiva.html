Bloco 2 · Processo

QScrum

/ 03
O que é
  • Scrum + um papel a mais: o Quality Manager.
  • Mesmo ritmo: sprints semanais, daily assíncrona, planning e review.
  • Cada US tem critério BDD escrito antes do código.
Dev entrega→QM valida BDD→Merge
Por que muda o jogo
  • QM valida independente de quem implementou.
  • Bug pego antes do merge não chega em produção.
  • Testes automatizados são condição de aprovação, não opcional.
  • BDD vira linguagem comum entre PO, Dev e QM.

A dinâmica real das sprints

/ 05

Bloco 3 · Construção

As primeiras decisões

/ 07

Python 3.13 + Flask

A equipe já conhecia Python; Flask cresce sem overhead de framework pesado.

MySQL sem ORM

SQL direto, connection pool manual — entender o banco de verdade, sem abstração.

Railway (CD)

Cada push = produção atualizada; sem infraestrutura manual.

Jinja2 + CSS puro

Sem React, sem SPA — templates server-side, simples e eficazes.

Bloco 3 · Construção

O que foi fácil, o que travou

/ 08
Foi fácil
  • Auth e CRUD de usuários — padrões bem estabelecidos em Flask
  • Deploy contínuo desde a Sprint 1 — Railway funcionou de primeira
  • Blueprints escalaram para 7 domínios sem dor
  • Testes de backend em pytest — setup simples com fixtures
Travou
  • US14/US15 ficaram numa branch de backup por 2 sprints — pego pelo QM
  • Schema drift: ALTER TABLE manual sem migration tool
  • Conflito de branches: 3 devs no mesmo schema
  • Playwright (testes UI) aprendido do zero
Bloco 3 · Construção

O que funcionou no processo

/ 09

Validação paralela

BUG-01 a BUG-05 pegos pelo QM antes de produção. Horas de debugging economizadas.

Sprints curtas (1 semana)

Forçaram entregas reais — WIP infinito virou produto funcionando.

Deploy contínuo

Railway desde a Sprint 1 — PO sempre na versão mais recente. Feedback real, não de demo local.

Bloco 3 · Construção

O que não funcionou

/ 10

Branch isolada

US14/US15 ficaram em backup/sprint3-willian por 2 semanas sem merge. Portadas na Sprint 5.

Schema sem migration

ALTER TABLE manual em produção, sem Alembic/Flyway. Inconsistência local/prod em dois momentos.

Comunicação assíncrona

Decisões de banco chegavam às 23h no grupo. O dev do dia seguinte não sabia da mudança.

Testes UI tardios

Playwright escrito no fim. US21–26 com cobertura de backend, mas não de UI — dívida técnica.

Bloco 5 · Qualidade

Critérios BDD

/ 12
Cenário: Aluno agenda slot disponível Dado que estou logado como aluno E existe slot de "Cálculo I" às "14:00" Quando clico em "Agendar" Então vejo "Agendamento confirmado" E o slot aparece como "Agendado"
  • O PO escreve o critério antes da US entrar em dev
  • O Dev implementa para satisfazer o cenário
  • O QM testa o cenário e escreve o teste automatizado
  • Divergência = bug registrado, US não aprovada

Uma tarefa bem escrita

/ 13
● Closed#19[US14] Aluno cancela um agendamento
User story
Como aluno, quero cancelar um agendamento, para liberar o horário se não puder ir.
Definition of Ready
✔Formato Como / Quero / Para que
✔Critérios de aceitação definidos
✔Dependência clara — depende de US12
✔Estimada — 3 pontos (Planning Poker)
Definition of Done
✔Código atende aos critérios
✔Regra das 6 horas testada
✔Sem flags pendentes do QM
✔QM atualizou o Sprint Tales
Critérios de Aceitação · BDD
Cenário 1 — cancelamento ok
Given aluno com agendamento confirmado When cancela com +6h de antecedência Then agendamento removido e o horário volta a aparecer como disponível
Cenário 2 — fora do prazo
Given agendamento em menos de 6h When aluno tenta cancelar Then rejeita: "Cancelamento não permitido com menos de 6 horas de antecedência"

28/05 · QM Review

🚩 Flag QM — implementação incorreta

O endpoint cancelava a presença numa sessão votada, não o agendamento individual. O horário não voltava para DISPONIVEL — o Cenário 1 não era atendido.

Movido para Ready for Review, pendente de correção antes de Done.

W
Willian · QM
28/05 · Corrigido e fechado

✓ Corrigido e fechado

Novo endpoint POST /agenda/agendamentos/<id>/cancelar: valida o dono e a regra das 6h, cancela o agendamento e reverte a disponibilidade na mesma transação (SELECT FOR UPDATE).

Frontend ganhou a seção "Meus agendamentos" com botão Cancelar. Critérios atendidos.

O mesmo fluxo da Validação Paralela — aqui registrado no board, de verdade.

Bloco 6 · Aprendizados

Aprendizados técnicos

/ 16

Flask sem ORM

SQL direto obriga entender o banco: joins explícitos, transações manuais, sem N+1 escondido.

MySQL em produção

Schema drift é real — sem migration tool vira dívida. Próximo projeto: Alembic na Sprint 0.

Railway como CD

Env vars, MySQL gerenciado, deploy por push. Produção desde a Sprint 1 sem ops manual.

Jinja2 + CSS puro

SSR é subestimado. Sem bundler, sem transpilação — template simples, resultado limpo.

Bloco 6 · Aprendizados

Aprendizados de processo

/ 17
O que levamos
  • Validação paralela precisa começar na Sprint 1 — não funciona se o QM entra no fim
  • Testes de UI devem acompanhar a US, não vir depois
E mais
  • Migration tool desde o início — schema sem Alembic dói cedo
  • Feature branch por US, não por dev
  • BDD antes do código muda a conversa — para de adivinhar o que "funcionar" significa
